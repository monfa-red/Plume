# Plume ‚Äî Implementation Plan

This document is the build plan for Plume v1 in Rust. `SPEC.md` is the source of truth for what to build; this file is how we build it.

---

## Goal

A single binary, `plume`, that compiles `.plume` files to SVG per `SPEC.md`. Production-quality base ‚Äî clean code, modular, no `unsafe`, good tests, easy to read. Tiny edge cases can land later; the structure shouldn't need rework.

## Architecture

Five-phase compiler pipeline, one module per phase:

```
.plume text ‚Üí [lex] ‚Üí tokens ‚Üí [parse] ‚Üí AST ‚Üí [resolve] ‚Üí resolved IR
                                                              ‚îÇ
                                                              ‚ñº
                                                 [layout] ‚Üí positioned tree
                                                              ‚îÇ
                                                              ‚ñº
                                                 [route]  ‚Üí wire paths
                                                              ‚îÇ
                                                              ‚ñº
                                                 [render] ‚Üí SVG text
```

Each phase has a clear contract:
- `lex(&str) -> Result<Vec<Token>, Error>`
- `parse(&[Token]) -> Result<File, Error>`
- `resolve(File) -> Result<Program, Error>`
- `layout(&Program) -> Result<LaidOut, Error>`
- `route(&LaidOut) -> Result<Routed, Error>`
- `render(&Routed) -> Result<String, Error>`

Every phase is fallible. Some ¬ß15 errors are inherently layout-time (`col=5 exceeds cols=3`, `col-widths` length mismatch, auto-size with neither dims nor text on a primitive that requires them); pre-validating them in resolve would duplicate work. No panics in the library path ‚Äî surface invariants as errors.

## Locked decisions

These are firm; don't revisit during implementation.

| Decision | Rationale |
|---|---|
| Single crate (not workspace) | 3000‚Äì5000 LOC. Split later if WASM or LSP targets land. |
| Hand-written recursive-descent parser | Spec is LL(1) (¬ß16). Best error messages, easiest to debug, no parser-combinator overhead. ~600 LOC per spec note. |
| Hand-written SVG output via typed builder helpers | Full control of formatting (matters for snapshot tests); zero deps; tiny surface (~12 element types). |
| No `unsafe` anywhere | Not needed for any part of this. |
| `clap` (derive) for CLI args | De facto standard. |
| `insta` for snapshot tests | Best DX for output-comparison. |
| LSP-format errors as plain `Display` impls | Per spec ¬ß15. Stay manual; only reach for `ariadne`/`miette` if errors feel rough. |
| Material Symbols bundled at build time | Apache 2.0 license. `build.rs` scans `assets/icons/` and emits a static path map. |
| Embedded char-width table for default font | Per spec ¬ß6 ‚Äî guarantees reproducible layout across hosts. |

## Open decisions

Decide at the moment they bite. Listed here so future-me knows they're not settled.

| Question | Options | Notes |
|---|---|---|
| Icon table shape | `phf::Map` (build-time perfect hash) / `&[(&str, &str)]` sorted slice with binary search / `match` arm | Pick when implementing `render/icons.rs`. `phf` is most elegant; binary search has zero deps. |
| Logging | `tracing` / `log` / just `eprintln!` for `--check` diagnostics / none | Reach for one when debug output is actually needed. |
| Color parsing | Hand-write (~50 LOC) / `csscolorparser` crate | Spec set is small (hex 3/6/8, named, rgb/rgba/hsl, var, none). Lean hand-write but reassess if it sprawls. |
| Theme CSS parsing | Hand-write line scanner / `cssparser` / `lightningcss` | We only extract `--plume-*: value;` lines. Hand-write probably wins. |
| Embedded char-width table source | Generate from a known font's metrics / hand-curated averages / single-width approximation | Start with the simplest that gives reasonable layout; upgrade if text wraps wrong. |
| Icon set scope | All ~3000 Material Symbols (few MB binary) / curated subset (~100‚Äì200) / dynamic load | Start with whatever ships easiest; tree-shake the output regardless. |
| Workspace split | Single crate now / split into `plume-core` + `plume-cli` later for WASM/LSP | Defer until a second consumer appears. |
| Error pretty-printer | Plain LSP format (spec-compliant) / `ariadne` for rich spans / `miette` | Start plain. Upgrade only if user feedback demands. |

## Sprint plan

Each sprint produces a working, committed artifact. Sprints are sized for ~2‚Äì5 days of focused work; adjust as needed.

### Sprint 0 ‚Äî Skeleton + MVP (1 day)

**Goal:** Smallest possible end-to-end pipeline. Validates the whole architecture before specializing.

**Deliverables:**
- `cargo new --bin plume` scaffolded.
- `Cargo.toml` with project metadata + locked deps.
- `rustfmt.toml`, `clippy.toml`, `.gitignore`, `LICENSE`, `README.md`.
- GitHub Actions CI: `cargo build`, `cargo test`, `cargo clippy --all-targets -- -D warnings`, `cargo fmt --check`.
- Module skeleton (empty `lexer.rs`, `parser.rs`, etc. with stub functions).
- One end-to-end test: `scene { :rect "hello" }` compiles to a valid SVG with a `<rect>` and `<text>`. Snapshot via `insta`.
- `resvg` CLI installed; PNG render saved next to SVG snapshot for visual smoke check.

**Definition of done:** `cargo run -- samples/hello.plume` writes valid SVG to stdout, opens in a browser as expected.

### Sprint 1 ‚Äî Lexer + parser (3‚Äì5 days)

**Goal:** Parse any valid Plume file into AST per SPEC ¬ß16.

**Deliverables:**
- Full token set: idents, keywords (block names), sigils (`:`, `.`), operators (all 9 wire ops, `=`, `;`, `{`, `}`, `(`, `)`, `[`, `]`, `,`), literals (number, string, color), `var(`/`rgb(`/`rgba(`/`hsl(`, comments (skipped).
- Source spans tracked for every token.
- AST types in `ast.rs` mirroring EBNF rules 1:1.
- Recursive-descent parser handling all five blocks, node decls, wire decls, primitive decls, label sugar (node + wire), bracket endpoint anchors, bare attrs (fixed set).
- Errors with line:col per spec ¬ß15.
- Tests: round-trip parse for every spec example + invalid-input cases.

**Definition of done:** every example in `SPEC.md` parses without error; invalid examples produce errors at the right span.

### Sprint 2 ‚Äî Resolution (2‚Äì3 days)

**Goal:** Convert AST into a fully-resolved IR ‚Äî every reference looked up, every default applied.

**Deliverables:**
- Defaults table: built-in fallbacks ‚Üê `--theme FILE` (via `theme.rs`) ‚Üê `defaults {}` entries.
- **`VarKind` split ‚Äî first-class.** Every CSS variable is tagged `Layout` or `Visual` at registration time. Layout vars resolve to a concrete number/value during resolve so phase 3 can do math; visual vars stay as `var(--plume-name)` strings that emit verbatim. Using a `Visual` var in a layout-affecting attr is an error; using a `Layout` var in a visual attr resolves to its baked value but still emits the live reference so runtime CSS can theme it. Test: a `--theme` override of `--plume-gap` changes layout, while a runtime CSS override of `--plume-gap` does not (visual-only runtime path).
- **Ordered attr storage.** Inline attrs on a node, wire, or primitive are stored as an ordered `Vec<(Key, Value)>` *until* ¬ß12 merging finalizes them. Required because marker resolution (¬ß7) is source-order-sensitive: `marker=arrow marker-end=dot` and `marker-end=dot marker=arrow` produce different SVG. Merging collapses to a map only after the marker pass reads positions.
- Styles symbol table with composition (`.other`) and cycle detection.
- Shapes symbol table with `:base` resolution, attr merging, cycle + depth-16 detection.
- Scene tree with each node's `:type` and `.style`s resolved, attrs merged per ¬ß12 specificity.
- Wires list with referenced IDs resolved.
- Errors per spec ¬ß15 for all forward-ref / cycle / unknown cases.

**Definition of done:** the complete example in SPEC ¬ß20 resolves cleanly into the IR. All ¬ß15 error cases produce the documented messages. Var-kind and marker-order tests in place.

### Sprint 3 ‚Äî Layout (1 week ‚Äî hardest sprint)

**Goal:** Compute bboxes and positions for every node.

**Deliverables:**
- Bbox math for all 14 primitives including stroke contribution (¬ß7 rule 4).
- Auto-sizing: text + `text-pad` for shapes with text children, otherwise per-shape defaults from CSS vars (¬ß6 auto-sizing table).
- Flex layouts (`row`, `column`) with `h=`/`v=` alignment + distribution values.
- Grid layout: explicit `col`/`row`, declaration order auto-flow, `colspan`/`rowspan`, `col-widths`/`row-heights` (scalar + list).
- Anchor resolution: 9 inner + 8 `out-*` anchors with the "exclude out-* children from parent bbox" recursion fix (¬ß6).
- Multi-value `padding`/`gap`/`radius` (scalar / 2-val / 4-val).
- `at=(x,y)` removes from flow but still expands bbox.
- `rotation` applied last; rotated bounding rectangle propagates up.
- Embedded char-width table for text bbox.

**Definition of done:** SPEC ¬ß20 complete example lays out correctly ‚Äî every node has a sensible bbox and position. Visual PNG check matches expectations.

### Sprint 4 ‚Äî Wire routing (3‚Äì4 days)

**Goal:** Compute paths and marker positions for every wire.

**Deliverables:**
- Edge selection: bracketed anchor wins, else nearest-edge with tie-break right > bottom > left > top.
- L- or Z-bend route computation between source and target bboxes.
- Self-loop geometry per spec ¬ß9.
- Marker placement (`arrow`/`dot`/`diamond`/`crow`/`none`) with operator-driven defaults, source-order override per spec ¬ß7.
- Marker sizing: `max(arrow-head, thickness √ó 5)`. Inset 4 px from endpoint.
- Wire-label positioning at `start`/`mid`/`end`/`0..1` anchors with `offset` in local tangent frame.

**Definition of done:** all wires in SPEC ¬ß20 render with correct routes and markers.

### Sprint 5 ‚Äî SVG render (3‚Äì4 days)

**Goal:** Emit valid, clean SVG.

**Deliverables:**
- Document shell with `viewBox`, `class="plume"`, `<style>` containing `@layer plume.defaults { ... }`.
- Typed SVG element builders in `render/svg.rs` (Element, attr setters, child append, write to string, XML escape).
- Each primitive ‚Üí its SVG element(s). Diamond ‚Üí polygon, cloud ‚Üí templated path, cyl ‚Üí ellipses + lines, etc.
- Filter generation for `shadow` (one filter per unique parameter combo, de-duped in `<defs>`).
- `fit=clip` via `<clipPath>`.
- Material Symbols icon embedding ‚Äî tree-shaken: only referenced icons in `<defs>`.
- Auto-classes per spec ¬ß13: `plume-node plume-shape-{type} plume-shape-{parent} plume-style-{name}`.
- Wire rendering with `plume-marker plume-marker-{type}`.
- `--no-defaults` mode that omits the `@layer` block.

**Definition of done:** SPEC ¬ß20 example produces a valid SVG that renders correctly in `resvg` and a browser. Snapshot tests cover every primitive and template.

### Sprint 6 ‚Äî CLI + polish (2‚Äì3 days)

**Goal:** Production-ready CLI.

**Deliverables:**
- Full `clap` config: `-o`, `--format`, `--standalone`, `--no-defaults`, `--check`, `--theme`, `-h`, `-V`.
- stdin support (`plume -`).
- Exit codes per spec ¬ß14.
- `--check` mode runs through resolve, no render.
- Error display polish: file:line:col format, helpful context.
- README with install, basic usage, link to SPEC.

**Definition of done:** the CLI matches spec ¬ß14 in all flags and exit codes.

### Sprint 7 ‚Äî Conformance suite + docs (ongoing)

**Goal:** Bulletproof coverage and a great first-run experience.

**Deliverables:**
- `samples/` directory with one `.plume` per spec feature (~50‚Äì80 files).
- Conformance test: walk `samples/`, compile each, snapshot SVG via `insta`.
- For a curated subset, render to PNG and visually verify.
- `docs/` mdbook scaffolding with tutorial + recipes (can be deferred to a follow-up).
- README polish.

**Definition of done:** every spec feature has at least one sample exercising it. CI runs the full conformance suite.

## Testing strategy

Layered:

1. **Unit tests per module** ‚Äî pure logic that doesn't need integration. E.g., bbox math, color parsing, anchor resolution.
2. **Snapshot tests with `insta`** ‚Äî phase outputs (token stream, AST, resolved IR) and final SVG strings. `insta review` for accepting changes.
3. **Conformance tests** ‚Äî `samples/` files compiled and snapshotted.
4. **Visual checks** ‚Äî for layout-sensitive samples, render to PNG via `resvg` CLI and inspect with the multimodal `Read` tool. No human required for the common case.
5. **Property tests** (optional, later) ‚Äî `proptest` for parser robustness.

CI runs everything on push.

## Visual verification loop

For sprints where the output is visual:

```
1. Edit code.
2. cargo run -- --bake-vars samples/<sample>.plume > /tmp/out.svg
3. resvg /tmp/out.svg /tmp/out.png
4. Read /tmp/out.png  (see the rendered image)
5. Compare against expectation. Iterate.
```

Install once:
```bash
cargo install resvg
```

**Sprint 0 finding ‚Äî `--bake-vars` is required for the loop.** Both `resvg` and `librsvg`'s `rsvg-convert` ignore CSS custom properties: a spec-conformant SVG with `fill="var(--plume-fill)"` and an `@layer plume.defaults { ... }` `<style>` block renders as black-on-black (vars fall back to `black`, the default for `fill`/`color`). Browsers render the same SVG correctly. We need a CLI/library flag (working name `--bake-vars`) that emits literal colors instead of `var()` references for static rendering ‚Äî useful both for our visual test loop and for SVG consumers that don't support CSS (email clients, image libs, etc.). Implement in Sprint 5 alongside the renderer. Layout vars already bake at compile time; this only changes visual var emission.

For automated regression: `tests/conformance.rs` does steps 2‚Äì4 for every sample and fails if a snapshot diverges.

## Notes on style

- One concept per file. Split modules when they cross ~500 LOC.
- Public API on `lib.rs` is small: `compile_str(&str) -> Result<String, Error>` and `compile_file(&Path) -> Result<String, Error>`. Everything else is `pub(crate)` or private.
- Errors implement `Display` in LSP format. No panics in the library path; the CLI converts errors to exit codes.
- Builder-style SVG helpers should read like the SVG they produce:
  ```rust
  g().class("plume-node plume-shape-rect")
     .transform_translate(x, y)
     .child(rect().w(100).h(40).fill("white"))
     .child(text("Drive PSU").at(0, 0))
     .write_to(&mut out);
  ```
- Prefer `match` over chains of `if let`. Prefer named structs over tuples for anything with > 2 fields.
- No comments explaining what code does; if it's not obvious, rename the thing.
